# Demo Data Management — Design Spec

## Problem
The system has no demo/training data. New installations and admin demos require manually entering records across all collections. Admins also need a way to wipe all data clean for a fresh start.

## Solution
A "Demo Data" section in the System Settings page (`/settings`, already admin-only) with two capabilities:

### 1. Seed Demo Data
Populates all collections with ~300 realistic Philippine barangay records using deterministic hardcoded data arrays.

### 2. Erase Data
One-click erase of selected collections (with ConfirmDialog) in safe dependency order. Admins pick which collections to clear via checkboxes + a "Select All" toggle.

## File Changes

### New file: `src/features/settings/demoData.ts`
- `AVAILABLE_COLLECTIONS` — list of `{ id, label }` for all 10 collections
- `seedCollections(ids, onProgress)` — seeds in dependency order using existing API create functions
- `eraseCollections(ids, onProgress)` — deletes in reverse dependency order; graceful fail on activity_logs 403

### Modified: `src/features/settings/SystemSettings.tsx`
- New card at the bottom with:
  - Checkbox list + Select All
  - "Seed Demo Data" button (with progress log)
  - "Erase Selected" button (with ConfirmDialog + progress)
- Progress area showing status messages + final summary

## Seed Data Volumes (~300 total)
- Households: 24
- Residents: 120
- Documents: 50
- Blotter: 20
- Assets: 30
- Visitors: 40
- Meetings: 10
- Agenda Items: ~40 (3-5 per meeting)
- Calendar Events: 16
- Activity Logs: auto-generated by API create calls

## Seed Order (dependency-respecting)
1. households → 2. residents → 3. meetings → 4. agenda_items → 5. calendar_events → 6. document_requests → 7. blotter_records → 8. assets → 9. visitor_logs

## Erase Order (reverse dependency)
1. agenda_items → 2. calendar_events → 3. document_requests → 4. residents → 5. meetings → 6. households → 7. blotter_records → 8. assets → 9. visitor_logs → 10. activity_logs (graceful — PB rules may reject)

## Data Sources
Hardcoded arrays of:
- Filipino surnames & given names (gender-appropriate)
- Purok values from system defaults
- Barangay document types, purposes, statuses
- Blotter incident types & realistic dispute narratives
- Barangay asset types (tables, chairs, sound system, etc.)
- Visitor purposes
- Meeting types & agenda topics
- Calendar event types

## UI
- Matches existing System Settings card styling
- `ConfirmDialog` for erase (destructive variant)
- Inline progress log for both seed and erase
- Final summary: "Seeded 120 residents, 24 households..." / "Erased 6 collections"
